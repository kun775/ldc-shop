# `_workers_next` 商品签到积分抵扣设计

## 1. 背景

当前 `_workers_next` 的下单逻辑里，只要用户在购买弹窗勾选了积分抵扣，服务端就会按 `1 积分 = 1 金额单位` 直接抵扣订单金额，最高可以把订单抵到 `0`。

这带来两个直接问题：

- 商品本身没有“是否允许签到积分抵扣”的配置，所有商品都默认可抵。
- 购买弹窗会在用户有积分时默认勾选积分抵扣，导致默认支付路径不是 `credit.linux.do`，而是“先尽量把积分抵光”。

这和当前运营需求不一致。新的目标是：

- 在 `/admin/product/edit` 为商品增加签到积分抵扣配置。
- 抵扣规则改为“最多抵扣订单金额的百分比”。
- 默认支付方式恢复为 `credit.linux.do` 授权扣款，积分抵扣只作为商品级可选能力。

## 2. 目标

本次改造只针对 `_workers_next`，目标如下：

- 后台商品编辑页支持配置“是否开启签到积分抵扣金额”和“最多抵扣百分比”。
- 前台购买弹窗只在商品允许时展示积分抵扣选项。
- 积分抵扣不再默认开启，用户需要主动勾选。
- 服务端按商品配置计算积分最大可抵扣额度，禁止再出现默认 100% 抵扣。
- 非 0 元订单默认继续走 `credit.linux.do` 支付；只有商品明确支持且用户积分足够时，才允许积分全额抵扣。

## 3. 范围

### 3.1 本次范围内

- 商品表新增积分抵扣相关字段。
- 商品编辑页新增配置项。
- 商品保存逻辑新增字段校验与兼容补列。
- 购买弹窗根据商品配置展示积分抵扣说明与结算结果。
- 下单服务端重算积分抵扣额度，调整订单金额与 `points_used` 记录逻辑。

### 3.2 明确不在本次范围内

- 不改造积分账本表结构。
- 不增加全局积分抵扣设置。
- 不改动支付网关接入方式。
- 不新增优惠券、满减、会员折扣等其他价格体系。

## 4. 推荐方案

采用“商品级开关 + 百分比上限 + 服务端强校验”的方案。

核心规则：

- 每个商品单独配置是否允许签到积分抵扣。
- 每个商品单独配置“最多抵扣订单金额的百分比”，例如 `10` 表示最多抵订单金额的 `10%`。
- 前台默认不勾选积分抵扣。
- 服务端永远以商品配置为准重新计算，不信任前端传入的展示结果。

选择该方案的原因：

- 完全贴合当前需求，不需要引入全局设置。
- 商品维度可控，方便按商品差异化运营。
- 对现有支付链路改动最小，主要集中在结算前的金额计算。

## 5. 数据设计

### 5.1 商品表新增字段

在 `products` 表新增两个字段：

- `point_discount_enabled`：是否开启签到积分抵扣，布尔值，默认 `0`
- `point_discount_percent`：最多抵扣订单金额百分比，整数，默认 `0`

Drizzle 字段命名对应为：

- `pointDiscountEnabled`
- `pointDiscountPercent`

### 5.2 默认值与兼容策略

- 老商品默认视为：
  - `point_discount_enabled = 0`
  - `point_discount_percent = 0`
- 保存商品时沿用当前项目已有的“缺列时自动补列”策略，通过 `ALTER TABLE` 兼容 Cloudflare D1 上的历史表结构。

### 5.3 字段校验

服务端保存商品时执行以下规则：

- 未开启积分抵扣时：
  - `pointDiscountEnabled = false`
  - `pointDiscountPercent = 0`
- 开启积分抵扣时：
  - `pointDiscountPercent` 必须是整数
  - 取值范围限制为 `1-100`
- 非法输入直接拒绝保存，不做静默纠正

## 6. 后台商品编辑页设计

修改页面：`/admin/product/edit/[id]`

在商品价格相关区域新增一个积分抵扣配置卡片，包含：

- 复选框：`开启签到积分抵扣金额`
- 数字输入框：`最多抵扣订单金额百分比`
- 辅助说明：
  - `10` 表示最多抵订单金额的 `10%`
  - 未开启时默认全部走 `credit.linux.do` 支付

交互规则：

- 未勾选时，百分比输入框隐藏或禁用。
- 编辑已有商品时，回显当前配置。
- 新建商品默认关闭积分抵扣。

## 7. 前台购买弹窗设计

修改组件：购买弹窗与下单按钮相关组件。

### 7.1 展示规则

只有同时满足以下条件时，才显示“使用签到积分抵扣”选项：

- 用户已登录
- 商品 `pointDiscountEnabled = true`
- 商品 `pointDiscountPercent > 0`
- 用户当前积分大于 `0`
- 当前订单按比例计算出的可抵扣积分上限大于 `0`

否则不显示积分抵扣复选框。

### 7.2 默认行为

- 购买弹窗打开时，积分抵扣默认不勾选。
- 默认展示原价与应付金额一致。
- 用户主动勾选后，才展示抵扣后的应付金额。

### 7.3 前端说明文案

弹窗中需要明确展示：

- 当前可用积分
- 本商品最多可抵扣的积分上限
- 勾选后本次实际会使用多少积分
- 剩余仍需通过 `credit.linux.do` 支付的金额

这样可以减少用户对“为什么有积分却没有全额抵扣”的误解。

## 8. 结算与支付逻辑

### 8.1 订单金额计算规则

订单原价：

- `orderAmount = Number(product.price) * quantity`

当用户勾选积分抵扣且商品允许抵扣时，服务端按以下规则计算：

- `maxDiscountAmount = orderAmount * pointDiscountPercent / 100`
- 因积分为整数，取：
  - `maxDiscountPoints = floor(maxDiscountAmount)`
- 实际使用积分：
  - `pointsToUse = min(currentPoints, maxDiscountPoints)`
- 最终应付金额：
  - `finalAmount = max(0, orderAmount - pointsToUse)`

### 8.2 关键约束

- 不管前端如何展示，服务端都必须重新读取商品配置并重新计算。
- 商品未开启积分抵扣时，即使前端传了 `usePoints=true`，服务端也必须按 `0` 积分处理。
- 商品开启但比例过小导致 `maxDiscountPoints <= 0` 时，也视为本次不可抵扣。

### 8.3 默认支付路径

- `finalAmount > 0`：
  - 一律创建待支付订单
  - 一律走 `credit.linux.do` 支付参数生成逻辑
- `finalAmount = 0`：
  - 仅在商品配置允许且积分足够覆盖时成立
  - 保留现有“积分全额抵扣直接出单”的零元订单流程

这意味着默认场景下，订单仍然以 `credit.linux.do` 为主支付方式，积分只是抵扣部分金额。

## 9. 订单与积分记录

订单表现有字段继续沿用：

- `orders.amount`：记录实际应付金额
- `orders.points_used`：记录本次实际使用的签到积分

行为要求：

- 只记录实际使用的积分，不记录“理论可抵扣上限”。
- 若用户未勾选积分抵扣，则 `points_used = 0`。
- 若订单最终走支付网关，支付成功后不需要额外改写积分抵扣比例相关数据。

## 10. 实现影响面

预计涉及以下区域：

- 商品表 schema
- 商品保存 action
- 商品编辑表单
- 购买弹窗组件
- 下单 action
- 对应中英文本地化文案

本次改动不应影响：

- 非登录用户浏览商品
- 不支持积分抵扣的商品下单
- 现有支付回调与发货流程

## 11. 风险与处理

### 11.1 历史商品没有新字段

处理方式：

- 保存商品时自动补列
- 未配置的历史商品按“关闭积分抵扣”处理

### 11.2 前后端金额不一致

处理方式：

- 服务端重新计算，前端只做展示
- 前端展示口径与服务端计算公式保持一致

### 11.3 小额订单按比例后无法抵扣

例如：

- 订单金额 `9.9`
- 抵扣比例 `10%`
- 最多可抵金额 `0.99`
- 转成整数积分上限后为 `0`

该场景下本次不允许抵扣，保持原价支付。这是当前“积分只能按整数使用”模型下的自然结果，不额外引入小数积分。

## 12. 测试与验证

至少覆盖以下场景：

### 12.1 商品保存

- 新建商品，默认关闭积分抵扣
- 开启后填写合法比例并保存成功
- 开启后填写 `0`、负数、非整数、超过 `100` 时保存失败

### 12.2 购买弹窗

- 商品未开启积分抵扣时不显示积分选项
- 商品开启但用户无积分时不显示积分选项
- 商品开启且用户有积分时显示积分选项，但默认不勾选
- 勾选后应付金额按比例正确变化

### 12.3 服务端下单

- 未开启商品即使传 `usePoints=true` 也不抵扣
- 开启 `10%` 时，实际抵扣不超过订单金额的 `10%`
- 开启 `100%` 且积分充足时可以产生零元积分订单
- 开启 `100%` 但积分不足时，剩余金额继续走 `credit.linux.do`

## 13. 最终结论

本次改造的核心不是“再加一个表单字段”，而是统一商品配置、前台展示和服务端结算口径：

- 商品决定是否允许签到积分抵扣
- 比例决定最多可抵扣多少订单金额
- 用户是否使用积分由用户主动选择
- 默认支付主路径恢复为 `credit.linux.do`

这样可以在不推翻现有支付链路的前提下，精准收回“签到积分默认全额抵扣”的失控行为。
